# Holy Ghostwriter

_A Flask application to create Dutch-language sermons using OpenRouter LLMs._

- Make a Python Flask application called "Holy Ghostwriter" that uses a web interface (on localhost port 5010) to query an OpenRouter model for generating a Christian sermon about a bible text. The interface should be in Dutch. Use Bootstrap for layout and styling etc.
  - Though the name of the app suggests some playfulness with regard to sermonwriting, this is a serious business and there should be no other jokes regarding Christianity.
- Dutch translations: sermon is preek, bible text is bijbeltekst, content notes is inhoudsbeschrijving.
- The front-end should be simple to use and functional. Style it with minimalist design, using classic tones that convey a sense of rest and inspiration.
- On the backend, there should be an OpenRouter-powered connection to an AI chatbot. On the homepage, there should be a gear-icon button to open settings in a modal: the OpenRouter model to use should be the only setting for now. This settings interface should have a list of available models, which are saved in the backend, where one model is currently selected. There should be a button to add a new model, which opens another modal to enter the name and the OpenRouter slug of the model. Save these settings persistently in the backend, e.g. in a JSON-file.
  - Query OpenRouter automatically to show token costs in the list, for price comparison.
  - As the first and currently selected OpenRouter model, use "NVIDIA Nemotron 3 Super (free)" with slug `nvidia/nemotron-3-super-120b-a12b:free`.
  - For now, use the OpenRouter API key in `api_key.txt`. Later we'll put this in environment variables.
- The folder `skills` contains Markdown files describing APIs and useful resources to implement things. Use it when developing the app.
- Have the main screen list all projects (sermons). Each project should concern one sermon. From here there should be links to edit and delete existing sermons, and add new sermons.
- Whenever querying a chatbot, use structured outputs to ensure that the generated text or chat response is properly parsed and inserted into the interface and memory. Write these formats yourself. I suggest having a schema for each type of action (see below) and for generated chat messages.
- A sermon page should show: a title, a reference to the Bible (text field, i.e. "Matthew 4: 12-15", note that this should not assume a fixed syntax), a large textfield for the user to enter the content to write the sermon about, and a large textfield that will populare with the generated sermon. For the sermon, use a WYSIWYG editor, and make it use Markdown underneath. An export button should be available that opens a modal to show the final text as rendered Markdown, and have a "Download" button in the modal to download as DOCX document.
  - Next to the bible reference field there should be a fetch button ("Laden") which should query the AI model to give the text of the bible reference. A paragraph display should be inserted below the text field which is populated with what the chatbot returns.
- There should be a folder `prompts` which contains Markdown files that are as "system prompts" when using actions in a chat. All prompts should be in Dutch. Make a first version for each that the user can edit. All prompt should be editable in the settings modal.
  - The `standards_and_preferences.md` file should contain user preferences on how to generate the sermon, like the Bible translation to use and the length of the sermon in words.
  - The `chat_personality.md` file should contain system-level instructions to the chatbot for how to behave in general towards the user. In the initial version, make this a friendly but not sycophantic assistent-like personality.
  - The `generate_sermon.md` file should contain instructions for generating sermons from bible references and content notes.
  - The `sermon_style.md` file should contain a description of the personal style of the preacher user using the system to generate sermons.
- The sermon page should have a left side with the aforementioned fields, and a right side that shows a chat interface. Each side should be in it's own container element that fills the screen vertically, and should scroll within that. The right side should show the chat with the AI. Track this chat (and all fields on this page) persistently for the sermon in the backend, as reopening the sermon page should restore the whole context. Display in typical messaging style, with right-aligned message bubbles for the user and left-aligned bubbles for the AI. The user can enter a message with a text field, but can also click action buttons above the message field, which should be made clickable (so always present, but not always "active") based on the current context:
  - If the bible reference field is populated, the content notes field is not populated, and the sermon field is empty, then activate the button "Genereer ideeën". This uses the `generate_ideas.md` prompt to query the AI to suggest themes and topics for a sermon about this bible text. These should only appear as a chat message from the AI.
  - If the bible reference field is populated, the content notes field is populated, and the sermon field is not, the button "Genereer preek" should be active. This should query the AI using the `generate_sermon.md` prompt. When this button is not active, show as a tooltip the conditions for making it active. Mark in the backend that the sermon has been generated. When the chatbot is finished responding, populate its response in the sermon field, then replace the "Genereer preek" button with a "Pas preek aan" knop.
  - The "Pas preek aan" button uses the `modify_sermon.md` prompt to query the AI to modify the sermon based on the current notes. It's response should replace the text in the sermon field.
- Whenever waiting for a response from the AI, show a message pending chat bubble, like done in apps like Signal. For example, use an ellipsis symbol where dots are pulsating in order, or something like that. Make sure to use streaming responses, so that text is populated in the field and/or message bubble when it arrives.
- Whenever receiving a response, make sure the schema separates between the content generated that should go into fields (e.g. the sermon) and the chatbot's guiding words like "I've generated this for you" etc. These should go in the chat as a message, while the generated content should be processed as described above.
- At the top of the chat window, show a header that gives the name of the chatbot and an "i" information icon, with the slug showing as a tooltip when the name or icon is hovered.
- When a button is clicked, this should be shown in the chat interface as a bubble with that button in it in a modified display state to indicate the action was used. Clicking it should show a modal that shows all the info sent to the chatbot. When a response comes back, the AI chat bubble should use a similar mark including modal to show that it generated, alongside the modifications to the form fields described above.
- Make sure that when the user leaves or closes the page, any outstanding requests are still processed. Maybe this means that requests are happening in the backend, not the frontend, I'll let you figure that out.
- Make the backend track the OpenRouter usage in dollars by using the usage returned in each response from OpenRouter. Track this usage per request, storing it with timestampts and associating with a sermon. On the sermon page, show the cost so far in the header, updating after a response message is completed.
- Find a way to package the app as a Windows and MacOS app, maybe using Electron or something similar. This app should be launchable from the start menu on Windows and Spotlight on MacOS, and attach to the taskbar on both OSes. It's context menu should have an open to "Open" which opens the app in a window that can show the web interface, and "Afsluiten" which should stop the webserver.
